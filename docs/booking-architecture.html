Booking architecture — what calls what

Doctra is the clinic's HIS (1C:Enterprise). Payload CMS is our editorial layer. The website talks to both, but only one is hot-path. Below: the actual API surface, where each call lives, and what changes when.

1. Doctra REST API — what the clinic exposes

All endpoints are POST to http://31.146.167.162:9090/doctra/hs/portal_exchange/v1/<method>. Implementation lives in src/lib/doctra-api.ts.

Doctra quirks (learned the hard way):
  • Every body must include "hash": "" — undocumented but required.
  • Auth header uses an underscore: Authorization: Bearer_<token>, not Bearer <token>.
  • Token TTL is 60 seconds — we cache + auto-refresh on status: 9.
  • get_time_slots with date_end = date + 1 returns ZERO slots — must pass at least date + 2 and filter client-side.
  • Slot dates are ISO (2026-04-28); booking submission needs compact (202604281300).

3. What happens when a patient opens /booking

  1. Page request hits the Server Component. src/app/(frontend)/[locale]/booking/page.tsx calls getBookingServicesFromPayload() + getBookingDoctorsFromPayload(locale) in parallel. Two Postgres queries. ~50ms total. Zero Doctra calls.
  2. HTML ships with the wizard already populated. BookingWizard mounts with initialServices + initialDoctors. Step 1 (service picker) and step 2 (doctor picker) render instantly — no spinner.
  3. Patient picks a doctor. DateTimeStep fires GET /api/booking/timeslots?summary=1&branchId=…&operatorId=…&dateBegin=…&dateEnd=…. This proxies to Doctra's get_time_slots for a 180-day window and returns the dates with availability. Live Doctra call — typically 500–2000ms. We use the result to grey-out unavailable days in the calendar.
  4. Patient picks a date. DateTimeStep fires GET /api/booking/timeslots?date=YYYY-MM-DD&branchId=…&operatorId=…. Single-day slot grid (with the 2-day workaround for Doctra's 1-day-window bug). Renders the time chips.
  5. Patient enters details and submits. POST /api/booking/submit → Doctra's booking_without_otac. Untouched by this work; do not test against the live endpoint.

Action Before After Why

Wizard renders services + doctors after second client fetch already in HTML SSR pre-loads. BookingWizard mounts populated.

Net effect: page paint went from "5+ second freeze" to instant. The only Doctra latency the patient ever feels is the calendar/slots fetch — and that only fires after they have actively picked a doctor, which is already a moment where some loading is acceptable.

6. Keeping Payload in sync with Doctra

Payload caches the catalogue, Doctra owns it. Sync is manual and on-demand:

  1. New doctor added in Doctra → admin signs into /admin, triggers POST /api/import-doctra (curl, button, or manual fetch from the admin UI).
  2. Import endpoint:
    • Fetches get_departments + get_doctors for each department.
    • For each Doctra doctor: if no Payload row matches doctraId, create with seeded name/specialty. If a row exists, only refresh the department link — editorial fields (photo, bio, qualifications) stay untouched.
    • Same for departments → Payload services.
    • Returns a per-collection summary: { created, updated, skipped, errors }.
  3. Admin opens the new doctor in Payload, uploads a photo, fills bio, polishes the Georgian/English name. Editorial work happens once, never blown away by future re-imports.
  4. Doctor leaves the clinic → admin checks the inactive checkbox. They disappear from the booking form and the website without losing the row (so old appointments still resolve).
Operational caveat: on a fresh database the booking page will be empty until someone runs the import. This is by design — Payload is the source of truth for what shows up on the site. In production, the inactive column also needs a one-time migration since push: true is disabled outside dev.
