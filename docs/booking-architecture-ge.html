ჩაწერის არქიტექტურა — რა რას ეძახის

Doctra არის კლინიკის HIS (1C:Enterprise) — სამედიცინო ინფორმაციული სისტემა. Payload CMS კი ჩვენი სარედაქციო ფენაა. ვებსაიტი ორივეს ესაუბრება, მაგრამ მხოლოდ ერთი მათგანია „ცხელი ბილიკი“. ქვემოთ — ფაქტობრივი API ზედაპირი, სად ცხოვრობს თითოეული გამოძახება და როდის რა იცვლება.

1. Doctra REST API — რას გვაძლევს კლინიკა

ყველა ენდპოინტი არის POST მისამართზე http://31.146.167.162:9090/doctra/hs/portal_exchange/v1/<method>. იმპლემენტაცია იწერება ფაილში src/lib/doctra-api.ts.

Doctra-ს თავისებურებები (გამოცდილებით ნასწავლი):
  • ყველა მოთხოვნის body-ში სავალდებულოა "hash": "" — დოკუმენტაციაში არ წერია, მაგრამ აუცილებელია.
  • Auth ჰედერი ქვედა ხაზით: Authorization: Bearer_<token>, და არა სტანდარტული Bearer <token>.
  • Token-ის ვადა 60 წამია — ვაქეშავთ და ავტომატურად ვაახლებთ status: 9-ის შემთხვევაში.
  • get_time_slots, თუ date_end = date + 1, აბრუნებს ცარიელ პასუხს — საჭიროა მინიმუმ date + 2, მერე ვფილტრავთ ჩვენ მხარეს.
  • სლოტების თარიღი ISO-შია (2026-04-28), ხოლო ჩაწერის გაგზავნისას უნდა იყოს კომპაქტური ფორმატით (202604281300).

ლაივ — არ ქეშირდება ნელი (~500–2000 მს)

არ შემოწმდეს ცოცხალზე წერს რეალურ ჩაწერებს

2. სად ზის Payload — სრული სურათი

ცხელი ბილიკი (ყოველი გვერდის ჩატვირთვისას / მომხმარებლის მოქმედებისას) Doctra-ს ლაივ გამოძახება (ნელი, მხოლოდ საჭიროებისას) ადმინის მიერ გაშვებული, მუშაობს ერთხელ იმპორტზე

3. რა ხდება, როცა პაციენტი ხსნის /booking-ს

  1. გვერდის მოთხოვნა Server Component-ში მიდის. src/app/(frontend)/[locale]/booking/page.tsx პარალელურად ეძახის getBookingServicesFromPayload()-ს და getBookingDoctorsFromPayload(locale)-ს. ორი Postgres-ის მოთხოვნა. ჯამში ~50 მწ. Doctra საერთოდ არ გამოიძახება.
  2. HTML მზა Wizard-ით იგზავნება. BookingWizard ჩაირთვება უკვე შევსებული initialServices-ით და initialDoctors-ით. 1-ლი ნაბიჯი (სერვისის არჩევა) და მე-2 ნაბიჯი (ექიმის არჩევა) მყისიერად ერქმევა — დატვირთვის ანიმაციის გარეშე.
  3. პაციენტი ირჩევს ექიმს. DateTimeStep ეშვება GET /api/booking/timeslots?summary=1&branchId=…&operatorId=…&dateBegin=…&dateEnd=…. ეს Doctra-ს get_time_slots-ისკენ მიდის 180 დღის ფანჯრით და აბრუნებს ხელმისაწვდომ თარიღებს. ლაივ Doctra გამოძახება — როგორც წესი 500–2000 მწ. შედეგად კალენდარში ვამქრქალებთ მიუწვდომელ დღეებს.
  4. პაციენტი ირჩევს თარიღს. DateTimeStep ეშვება GET /api/booking/timeslots?date=YYYY-MM-DD&branchId=…&operatorId=…. ერთი დღის სლოტები (Doctra-ს 1-დღიანი ფანჯრის ბაგის გვერდის ავლის მიზნით ვითხოვთ 2 დღეს). შემდეგ რენდერდება დროის ჩიპები.
  5. პაციენტი ავსებს მონაცემებს და ნიშნავს ჩაწერას. POST /api/booking/submit → Doctra-ს booking_without_otac. რეფაქტორმა არ შეეხო; ცოცხალ ენდპოინტზე არ მოწმდება.

4. ვის ეკუთვნის რომელი მონაცემი

მონაცემი ჭეშმარიტების წყარო როგორ ხვდება როგორ კითხულობს საიტი

ექიმების სია Payload doctors Doctra-ს get_doctors-დან /api/import-doctra-თი Postgres SELECT, ~50 მწ

6. Payload-ის Doctra-სთან სინქრონიზაცია

Payload აქეშავს კატალოგს, Doctra ფლობს მას. სინქრონი ხელით და მოთხოვნით ხდება:

  1. ახალი ექიმი დაემატა Doctra-ში → ადმინი შედის /admin-ში და უგზავნის POST /api/import-doctra-ს (curl-ით, ღილაკით ან მანუალურად ადმინ-პანელიდან).
  2. იმპორტის ენდპოინტი:
    • ეძახის get_departments-ს და get_doctors-ს ყოველი განყოფილებისთვის.
    • თითოეული Doctra-ს ექიმისთვის: თუ Payload-ში doctraId-ით არ მოიძებნა — აშენებს ახალს Doctra-ს მონაცემებით. თუ არსებობს — მხოლოდ კავშირს ანახლებს; სარედაქციო ველები (ფოტო, ბიო, კვალიფიკაცია) უცვლელი რჩება.
    • იგივე ლოგიკა მუშაობს Payload services-ისთვის.
    • აბრუნებს რეპორტს კოლექციების მიხედვით: { created, updated, skipped, errors }.
  3. ადმინი ხსნის ახალ ექიმს Payload-ში, ტვირთავს ფოტოს, ავსებს ბიოგრაფიას, აპრიალებს ქართულ/ინგლისურ სახელს. სარედაქციო სამუშაო კეთდება ერთხელ — მომავალი იმპორტი არასოდეს ანადგურებს.
  4. ექიმი მიდის კლინიკიდან → ადმინი მონიშნავს inactive ჩექბოქსს. ექიმი ქრება ჩაწერის ფორმიდან და საიტიდან, მაგრამ ჩანაწერი არ წაიშლება (ძველი ჩაწერები გრძელდება).
საოპერაციო ნიუანსი: სუფთა ბაზაზე ჩაწერის გვერდი ცარიელი იქნება, სანამ ვინმე არ გაუშვებს იმპორტს. ეს განზრახაა — Payload არის ჭეშმარიტების წყარო იმისა, რაც საიტზე გამოჩნდება. პროდაქშენში ასევე საჭირო ხდება inactive სვეტის მიგრაცია, რადგან push: true dev-ის გარეთ გამორთულია.
